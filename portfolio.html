No More Assets (November 2021)

This was my first game jam at the UW Whitewater game development club. If I recall correctly, the theme was 'absence', which I took full advantage of to focus mainly on programming. My main goal while coding the project was to investigate alternate methods of creating movement systems, which led me to tie my character's movement to an animation state machine. It was an interesting experiment, but it became obvious early on that as the complexity of the control system grew, it would get harder and harder to anticipate how the structure of the Animator Controller flows into each state from anywhere in the machine, meaning it was not easily scalable. Agrav Racer (April 2022)

This is a project that I submitted for my class coursework in MAGD 272. Normally, that course is intended as an introduction to Unity, with code and assets provided by the teacher in order to ease prospective developers into the process. At this point, I had been tooling around with Unity on and off for years and had been getting serious about it within the past several months, so I felt that I needed to up my game past what was required of me as a bare minimum. When most other people were making 2D platformers with premade sprites, I ended up building a top-down arcade racer. I got some help from a Youtube tutorial in order to set up the movement in a way that allowed drifting, but when the math of it was sorted out I tinkered around with it so that the driving style was to my liking. At the end of it, I had a set of 3 tracks where the best way to navigate through them quickly was to drive through turns sideways and down straightaways in reverse. I even had time to add multiple difficulty options to the AI racers, though their pathing is far from perfect.

Checkers (December 2022)

This project was more of a personal challenge to myself more than anything. I wanted to benchmark my progress in learning Csharp programming and the Unity system, so I gave myself the stipulation going in that I couldn't use tutorials to help me make this, and that I had to finish it within 24 hours from the start. Overall, it was a good test of my ability because Unity's toolkit is mainly oriented towards games that occur in real time rather than in turns. The solution I ended on was that each tile on the checkerboard was in one of several states, which were modifiable by player input and the state of neighboring pieces. On one hand, I made it happen within the constraints I gave myself and a game of checkers can be played in its entirety without any bugs. On the other hand, the code that controls the tile states is both monolithic and contains several instances of boilerplate that came back to bite me through development. For that reason, I consider this to be both an accomplishment and an important mistake to learn from.

Blacksite Theta (May 2023)

This was a semester-long capstone project carried out by UW Whitewater's MAGD 488 class, where the entire class of around two dozen students elect a game pitch from a pool of student-submitted options and spend the whole semester developing that idea. The concept we eventually settled on was actually the one put forth by my group, which resulted in me taking on a leadership role in the class. I had a heavy hand in the early concept, so I ended up being tasked as Narrative Lead while my friend Dayne Farris - who was also in the winning group - took charge on the large-scale team management. As such, my responsibilities during the development were to act as liason between him and the various 'departments' he set up, build out the game's story throughout its development, program the tools necessary to implement the story inside of the game (using Ink to translate between a storage file and my script that read out the individual dialog boxes), and offer assistance with completing coding tasks whenever possible. From our team of 8 programmers, I contributed roughly 10% of the existing codebase directly, and assisted with other programmers work whenever possible. Though our scope was likely a bit overlarge to begin with for a single-semester project with team members of somewhat variable levels of experience, the programming team never missed a prototype deadline and my other contributions followed suit.

Project DHPS, currently unfinished (July 2023)

A summer I've wanted to get back to at the start of this year, but have yet to do so due to responsibilities as the new Vice President for the UW game development club along with getting my ducks in a row for my eventual graduation. That being said, the heavy lifting has been already laid out within the project, containing my most technically involved codebase to date. In it, I've created a random dungeon generator, an AI designed to autonomously maneuver through this random dungeon to find a specific object and return to the entrance without knowing the position of the object they're looking for, and some AI minions for the player to use to keep them from doing so. The turnaround on completion of this first phase of the project was about 2 months of time in between working a 40+ hr/week summer job, though the project is entirely programmer art and has a few outlying bugs that need to be addressed once I get back to it.

The code for my dungeon generator. Essentially, it maps the area to a 2D array of ints and uses those states to determine how to place the various tiles of the map.

DungeonGraphGenerator.cs

Shown: generation of dungeon rooms, along with a brief demonstration of the RTS-like elements in place to facilitate use

An adventurer AI is spawned in search of the hoard. It receives a short list of random points on the Navmesh it can path to, and upon reaching the point closest to the objective it is notified and given the distance between itself and its objective. It uses this information to generate a new list of points within that radius, and continues to do so until it finds the objective within its line of sight sensors.
